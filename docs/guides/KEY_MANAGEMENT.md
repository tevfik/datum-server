# Key Management & Token Flow

This document outlines the architecture and workflows for authentication keys within the Datum Platform, specifically focusing on the distinction between User, Device, and Application keys.

## 1. Key Types & Conventions

The platform uses a prefix-based key system to instantly identify the type and handling logic for any given credential.

| Prefix | Name | Type | Generated By | Expiry | Purpose |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **sk_** | Static Key | Immutable | Server / Provisioning | Never | Simple devices (sensors, relays) without RTC/crypto capabilities. |
| **dk_** | Device Token | Dynamic (SAS) | Device Firmware | Configurable (e.g. 1d) | Secure devices (cameras, gateways) with Master Secret & RTC. |
| **ak_** | App Key | Immutable | User Dashboard | Never (User Revocable) | CLI (`datumctl`), CI/CD scripts, 3rd party backends. |
| **None** | User Token | JWT (Bearer) | Server (Login) | Short (e.g. 24h) | Frontend (React), Mobile App communication. |

### 1.1. Static Key (`sk_`)
- **Structure**: `sk_` + 32 hex chars.
- **Workflow**: 
    1. Assigned to device during **Provisioning**.
    2. Saved to device Non-Volatile Storage (NVS).
    3. Sent in `Authorization: Bearer sk_...` header.
- **Pros**: Simple, zero computation on device.
- **Cons**: If stolen, requires manual mutation/revocation.

### 1.2. Device Token (`dk_`) (SAS - Shared Access Signature)
- **Structure**: `dk_{unix_timestamp}.{hmac_signature}`
- **Workflow**:
    1. Device is provisioned with a **Master Secret** (never sent over wire after provisioning).
    2. Device calculates: `Sign = HMAC_SHA256(MasterSecret, "DeviceID:ExpiryUnix")`.
    3. Device constructs token: `dk_` + `ExpiryUnix` + `.` + `Sign`.
    4. Token is valid until `ExpiryUnix`.
- **Pros**: Master Secret never travels. Compromised token expires automatically.
- **Cons**: Requires RTC (Time) and HMAC crypto support on device.

## 2. Provisioning & Onboarding Flow

Which token method is used is decided **during the connection/provisioning phase** based on the Firmware's capability advertisement or the method chosen by the user in the Mobile App.

### Step-by-Step Flow

#### Phase 1: Handshake (Bluetooth / SoftAP)
1. **Mobile App** connects to **Device**.
2. **Device** advertises capabilities (e.g., `auth_mode: "static"` or `auth_mode: "rotating"`).

#### Phase 2: Registration (Server)
3. **Mobile App** calls Server API: `POST /devices/register`.
4. **Server** creates Device Record.
5. **Server** generates:
   - If `static`: A `sk_...` key.
   - If `rotating`: A random **Master Secret** (base64).

#### Phase 3: Configuration (Device)
6. **Mobile App** sends credentials to **Device**:
   - WiFi SSID/Pass
   - Device ID
   - **Auth Credential** (`sk_...` OR `Master Secret`).
7. **Device** saves these to NVS/EEPROM.
8. **Device** reboots and connects.

#### Phase 4: Operation
- **Static Device**: Sends `sk_...` in every request.
- **Rotating Device**: Wake up -> Sync Time (NTP) -> Generate `dk_` -> Send Request.

## 3. Security Guidelines for Developers

### Do NOT:
- Ever hardcode a `sk_` or `ak_` in frontend code (`.js` bundles).
- Use User JWTs (`eyJ...`) on IoT Devices. They expire and leave devices stranded.
- Share `dk_` Master Secrets.

### DO:
- Use `datumctl` to rotate compromised device keys immediately.
- Use TLS (HTTPS/MQTTS) for all traffic to prevent `sk_` sniffing.
- prefer Rotating Auth (`dk_`) for devices with internet access and RTC.
