# MQTT Hybrid Command Flow

This guide explains the "Hybrid Command Flow" architecture, which allows devices to be controlled via both **HTTP Polling** and **MQTT Subscription**, ensuring commands are accessible regardless of the device's connection protocol.

## Overview

Datum supports two primary device communication patterns:
1.  **MQTT Devices:** Maintain a persistent connection and receive commands instantly.
2.  **HTTP Devices:** Sleep to save power and periodically "poll" for pending commands.

**The Hybrid Flow ensures that a command sent via MQTT (e.g., from an external script, Node-RED, or physical button) is also persisted to the database, making it available for HTTP-only devices.**

## Architecture

### 1. Unified Command Ingestion
Whether a command comes from the REST API (`POST /dev/:id/cmd`) or an MQTT Publish (`dev/:id/cmd`), it is processed uniformly.

**Flow A: REST API (Web Dashboard, Mobile App)**
1.  User clicks "Toggle Relay".
2.  Server receives `POST /dev/:id/cmd`.
3.  Server saves command to DB ("pending").
4.  Server publishes to MQTT `dev/:id/cmd` (for live devices).

**Flow B: External MQTT Client (Node-RED, Physical Switch)**
1.  Switch publishes `{"action": "toggle"}` to `dev/:id/cmd`.
2.  **Datum MQTT Broker intercepts the message.**
3.  Broker checks if `command_id` already exists (Deduplication).
4.  If new, Broker saves command to DB ("pending").
5.  Broker continues publishing to subscribers (live devices receive it).

### 2. Deduplication Logic
To prevent infinite loops (API -> MQTT -> DB -> MQTT...), the broker checks the `command_id`:
-   **Known ID:** If the command was generated by the Server (Flow A), it has an ID known to the DB. The ingestion hook sees this and *skips* saving it again.
-   **Unknown/New:** If the command has a new ID (or no ID), it is treated as a new external command (Flow B) and saved.

## Usage Guide for External Clients

To control a Datum device via MQTT from an external tool:

**Topic:** `dev/<TARGET_DEVICE_ID>/cmd`
**Payload:**
```json
{
  "id": "optional-unique-id",
  "action": "toggle_relay",
  "params": {
    "state": true
  }
}
```

-   **id**: (Optional) Provide a UUID to ensure exactly-once processing. If omitted, the server generates one.
-   **action**: The action key defined in the device's Thing Description.
-   **params**: Parameters required by the action.

## Benefits
-   **Protocol Agnostic:** Control HTTP devices using MQTT tools.
-   **Offline Queuing:** If an MQTT device is offline when you publish, the command is saved. It can pull it from the DB (via HTTP poll) when it wakes up, or receive it via MQTT Retain (if configured).
-   **Unified Logs:** All commands, regardless of source, appear in the Device Command History.
